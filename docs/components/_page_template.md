## コンポーネントの節が扱う2つの軸

このセクションは次の2軸で構成される。両方とも独立に「なるべく網羅的に」扱う。

- **使い方の軸**: 各コンポーネントの仕様・使い方をAPIリファレンスよりユーザーフレンドリに説明する。
  これは継承可能かどうかと無関係に、全コンポーネントに必須。
- **拡張方法の軸**: 挙動を変える手段（継承、軽量フック、コールバック合成など）を説明する。
  「基底クラスがあり、ユーザーが継承する可能性があるものは例外なく記載する」という原則は
  この軸の話であって、「掲載するかどうか」を決める基準ではない
  （[[feedback_docs_page_split_criteria]]は独立ページ化の粒度を決める基準であり、
  「継承しないから載せない」という意味には使わない）。

## コンポーネントの3類型と、拡張系2節の書き方

対象コンポーネントは次の3類型のいずれかに当てはまる。類型に応じて「拡張フック」
「独自\<X\>の実装方法」の2節の中身を書き分ける（見出し自体は変えなくてよい）。

**類型A: 継承が拡張手段**（Crossover, Comparator, Surrogate等、これまでの大多数）
  - 「拡張フック」: `with_post`系ライフサイクルフックがあれば書く。無ければ節ごと省略する。
  - 「独自\<X\>の実装方法」: 基底クラスを継承し、抽象メソッドを実装する最小例を示す。

**類型B: 継承ではない拡張手段を持つ**（例: Termination — プレーンなcallableと
  `|`/`&`/`~`演算子による合成。Comparatorの`sorter=`/`dominator=`のような差し替え可能な
  Protocol引数も、差し替え対象自体は継承対象だが、それを受け取る側から見ればこの類型に近い）
  - 「拡張フック」の見出しは使わず、「\<X\>の拡張方法」のような見出しにしてよい。
  - 中身は「なぜ継承ではないか」（例: `TerminationCondition`は合成のための薄いラッパーで
    あり抽象基底ではない）を一言述べたうえで、実際の拡張手順（プレーンな関数を書く、
    演算子で合成する等）を示す。

**類型C: 使い方だけで、拡張手段を持たない、または限定的**（例: Population/Archive —
  基本はインスタンスとして使うAPIであり、通常は継承しない）
  - 「拡張フック」「独自\<X\>の実装方法」を無理に埋めない。この2節を
    「主要な属性とメソッド」のような使い方本位の見出しに置き換えてよい。
  - 限定的な差し替え点がある場合（例: `Algorithm.population_class`/`archive_class`を
    オーバーライドすれば独自Population/Archiveサブクラスに差し替えられる）は、
    それを短い注記として「関連コンポーネント」節の直前に置く程度に留める。
